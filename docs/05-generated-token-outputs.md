# 05 - Generated Token Outputs

## Output Philosophy

All output should come from canonical tokens. Do not generate CSS from source
files in one script and Mantine output from source files in another script.
That creates multiple sources of truth.

The generation flow is:

```txt
source fixtures -> canonical.json -> all generated outputs
```

## Output List

Generated files:

```txt
packages/tokens/dist/
```

## CSS Variable Output

CSS custom properties are the most portable output for React components and the
example app.

Recommended naming:

```txt
--ds-color-primitive-primary-10
--ds-color-text-default
--ds-color-background-body
--ds-space-md
--ds-radius-md
--ds-shadow-card
--ds-component-button-primary-high-background
--ds-component-button-height-md
--ds-font-size-heading-h1
--ds-line-height-heading-h1
--ds-font-weight-heading-h1
```

Example CSS:

```css
:root {
  --ds-color-primitive-primary-10: #cffafe;
  --ds-component-button-height-md: 36px;
  --ds-space-md: 8px;
  --ds-radius-md: 8px;
}

[data-mantine-color-scheme="light"] {
  --ds-color-text-default: #083344;
  --ds-color-background-body: #f8fafc;
  --ds-component-button-primary-high-background: #155e75;
  --ds-shadow-card: 0px 2px 8px 0px rgb(15 23 42 / 0.12);
}

[data-mantine-color-scheme="dark"] {
  --ds-color-text-default: #fdfdfd;
  --ds-color-background-body: #0f172a;
  --ds-component-button-primary-high-background: #0e7490;
  --ds-shadow-card: 0px 2px 8px 0px rgb(15 23 42 / 0.12);
}
```

## TypeScript Output

TypeScript output supports programmatic use and type-safe token names.

Example generated exports:

```ts
export const tokens = {
  color: {
    semantic: {
      text: {
        default: {
          cssVariable: "--ds-color-text-default",
          value: {
            light: "#083344",
            dark: "#FDFDFD"
          }
        }
      }
    }
  }
} as const;

export type TokenName =
  | "color.semantic.text.default"
  | "color.semantic.background.body"
  | "component.button.primary.high.background"
  | "space.md"
  | "radius.md"
  | "shadow.card";
```

Helper:

```ts
export function cssVar(name: TokenName): string {
  return `var(${tokenNameToCssVariable[name]})`;
}
```

## Metadata Output

Generate metadata that Storybook and CI can read:

```json
{
  "name": "demo-design-system-tokens",
  "source": "demo-design-token-fixtures",
  "tokenCount": 347,
  "modes": ["light", "dark"],
  "categories": ["color", "component", "radius", "shadow", "space", "typography"]
}
```

Avoid non-deterministic timestamps in committed metadata.

## Light And Dark CSS Files

Provide both combined and split CSS output:

```txt
tokens.css        # all tokens, mode selectors included
tokens.light.css  # only light-mode values
tokens.dark.css   # only dark-mode values
```

This helps different consumers:

- Storybook can import `tokens.css`.
- An app that handles mode itself can import the combined file.
- A static preview can import only one mode file.

## Documentation Data

Storybook token pages should not scrape CSS. Generate token docs data directly
from canonical JSON.

```txt
packages/tokens/dist/token-docs.json
```

Suggested shape:

```json
{
  "groups": [
    {
      "name": "color.semantic.text",
      "tokens": [
        {
          "name": "color.semantic.text.default",
          "cssVariable": "--ds-color-text-default",
          "value": {
            "light": "#083344",
            "dark": "#FDFDFD"
          }
        }
      ]
    }
  ]
}
```

## Quality Reports

Generate machine-readable and review-friendly quality reports:

```txt
packages/tokens/dist/token-quality.json
packages/tokens/dist/token-quality.md
```

The reports should cover:

- Token counts by category and type.
- CSS output coverage.
- Light/dark mode coverage for semantic colours, component colours, and shadows.
- Naming and CSS-variable validity.
- Source file coverage for mapped records and generated tokens.
- Warnings or errors from source mapping.

These reports are public demo artifacts and must be deterministic. They should
not read private raw exports or include absolute local paths.

## Generated File Headers

For generated TypeScript and CSS files, include a header:

```txt
Generated by @demo-ds/token-pipeline.
Do not edit directly.
Source: demo-design-token-fixtures.
```

For JSON files, put equivalent metadata in a `meta` object.

## Do Not Expose Source Paths By Default

Canonical tokens may contain source provenance during development, but public
package output should avoid exposing source filenames if that becomes noisy.

Recommended approach:

- Keep source provenance in `canonical.debug.json` for local debugging if needed.
- Keep public `canonical.json` clean.
- Do not include environment-specific path names in public docs.

## Snapshot Strategy

Snapshot the generated outputs that define the public contract:

- `canonical.json`
- `tokens.css`
- `token-names.js`
- Mantine theme token mapping

Avoid snapshotting files with timestamps, absolute paths, or
environment-specific output.

## Output Compatibility Promise

For the demo, treat these as public contracts:

- Token names.
- CSS variable names.
- Package exports.
- Mantine theme package exports.
- Component package exports.

Changing them should be deliberate and documented.
