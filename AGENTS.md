# AGENTS.md

## Repository mission

Build a public demo design-system pipeline from sanitised fixture tokens to canonical design tokens, generated CSS variables, generated TypeScript exports, a generated Mantine theme, a React component package, a Storybook site, and an example React app.

This repo is a demonstration of process and engineering quality. It must not contain work-specific, client-specific, confidential, proprietary, or brand-identifying source material.

## Hard safety rules

- Do not copy raw work token exports into this repository.
- Do not add files with private company names, internal brand names, private project names, or work system names.
- Do not add private fonts, logos, icon sets, screenshots, product copy, customer examples, or internal URLs.
- Do not add original unsanitised colour values or Figma metadata.
- Treat `packages/tokens/fixtures/sanitised-design-token-fixtures.zip` or its extracted contents as the only allowed token source.
- Generated output must be derived from the sanitised fixture only.
- Any test fixtures must be synthetic and generic.

## Engineering conventions

- Use TypeScript in strict mode.
- Prefer pnpm workspaces.
- Keep build logic inside packages rather than large root scripts.
- Prefer small, composable packages with clear public exports.
- Prefer deterministic generation. Generated output should be stable across repeated runs.
- Mark generated files with a header such as `Generated by @demo-ds/token-pipeline. Do not edit directly.`
- Validate all inputs before generating outputs.
- Keep public package names generic, for example `@demo-ds/tokens`, `@demo-ds/mantine-theme`, and `@demo-ds/components`.

## Expected commands

At minimum, keep these root commands working:

```sh
pnpm build
pnpm test
pnpm lint
pnpm typecheck
pnpm format:check
```

When changing token pipeline code, run:

```sh
pnpm --filter @demo-ds/token-pipeline test
pnpm --filter @demo-ds/tokens build
```

When changing theme or component code, run:

```sh
pnpm --filter @demo-ds/mantine-theme test
pnpm --filter @demo-ds/components test
pnpm --filter @demo-ds/storybook build
```

When changing the example app, run:

```sh
pnpm --filter @demo-ds/example build
```

## Implementation style

- Keep PRs/task outputs small and reviewable.
- Add tests in the same change as implementation code.
- Do not silently change generated snapshots. Explain why snapshots changed.
- Prefer accessible components by default.
- Do not wrap every Mantine component. Create wrappers only when they encode design-system decisions.
- Use semantic token names in application and component CSS. Avoid hardcoded colours in components.

## Token naming rules

- Canonical token names use lowercase kebab-case path segments.
- Mode names are `light` and `dark`.
- Primitive colours live under `color.primitive.*`.
- Semantic colours live under `color.semantic.*`.
- Spacing lives under `space.*`.
- Radius lives under `radius.*`.
- Typography lives under `typography.*`.

## Done means

A task is done only when:

- Code compiles.
- Relevant tests pass.
- Generated outputs are deterministic.
- Public APIs are documented.
- No forbidden source markers are introduced.
- The change fits the package boundaries described in `docs/01-target-repo-structure.md`.
