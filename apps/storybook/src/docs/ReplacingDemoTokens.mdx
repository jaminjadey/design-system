# Replacing Demo Tokens With Brand Tokens

For local testing, put private raw exports here:

```txt
.private/design-system/
```

This repo also ignores `tokens/raw/` and `tokens/imported/` for teams that use
that local convention. These paths are guarded so force-added private files fail
CI.

Run the raw importer:

```sh
pnpm tokens:import -- --input .private/design-system --output .private/normalised-token-output --config .private/design-system/import.config.json
```

In a private work repo, point the output directory at the token source folder
used by the token package build. Then run:

```sh
pnpm tokens:build
```

Mapping rules live in `token-pipeline.config.json`. Change that file when raw
export group names, mode names, semantic categories, component categories,
component leaf-name patterns, spacing, radius, or typography naming conventions
differ from the demo.

For component variables, map raw files into `components/Light.tokens.json`,
`components/Dark.tokens.json`, and `components/Dimensions.tokens.json`, or point
the component colour config at the same normalised light/dark files used for
semantic colours. Use `canonical.components.categoryPrefixes` for clear
component groups and `canonical.components.leafPrefixes` for mixed groups where
the component name appears in the leaf token name. The public demo uses
synthetic component fixtures only.
