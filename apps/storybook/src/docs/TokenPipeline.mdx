# Token Pipeline

The token pipeline is deterministic and package-oriented:

1. Demo token JSON, or locally imported private raw-token output, is scanned before generation.
2. Source token JSON is parsed into flat source records.
3. Source paths are mapped into canonical token paths.
4. Canonical tokens are validated.
5. Package outputs are generated from canonical JSON.

The generated output source is `{metadata.source}` and the generated package
contains `{metadata.categories.join(', ')}` categories.

## Generated Artifacts

- `canonical.json`
- `tokens.css`
- `tokens.light.css`
- `tokens.dark.css`
- `index.js`
- `token-names.js`
- `metadata.json`
- `token-docs.json`

Storybook token pages import `token-docs.json`; they do not scrape CSS and do
not read source token files.

## Raw Export Import

For local private raw exports, put files under `.private/design-system/` and
configure `import.config.json` to map raw files to the normalised source files
used by the token package.

```sh
pnpm tokens:import -- --input .private/design-system --output .private/normalised-token-output --config .private/design-system/import.config.json
```

The importer strips source-tool metadata, rejects forbidden markers, resolves
aliases, and writes an `import-report.json` with file/path findings. Keep real
brand inputs and imported outputs under `.private/` in this public repo.

## Mapping Config

Canonical mapping rules live in `token-pipeline.config.json`.

Use it to configure source files, primitive colour path rules, semantic category
prefixes, mode names, spacing/radius indexes, typography properties, and the
unsupported-token policy.

## Reports

The import step writes `import-report.json`.

The token build writes `build-report.json` with source record counts, skipped
records, renamed canonical paths, missing semantic modes, generated files, and
warnings.
